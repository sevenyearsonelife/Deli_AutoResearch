# Deli_AutoResearch

> 一个面向**长时程自主任务**（数天到数周）的协议框架（Protocol Framework）。

Deli_AutoResearch **不是一份可安装的代码库**，而是一套经过实战检验的工程约定：规定如何持久化状态、如何检测停顿、如何分层设置守护机制，以及哪些约束规范着智能体（Agent）的行为。实现细节留给采用者的运行环境自行决定。

## 这是什么

一个让长时程自主任务**可靠运行、无需人工干预**的协议框架。它把「让 Agent 连续工作几天甚至几周」这件事，从「祈祷模型不出错」变成「靠工程脚手架兜底」。

适用场景：

- 学术论文写作（综述、调研）
- 长时程研究（多轮迭代、证据累积）
- 无人值守的批量实验运行
- 任何需要 Agent 持续数小时以上、不依赖用户实时反馈的任务

## 为什么需要它

长时间运行的代码 Agent 会反复出现三种失败模式，而它们的共同根因是**缺失工程脚手架**，而非模型能力不足：

| 失败模式 | 表现 | 后果 |
|---------|------|------|
| **认知循环**（Cognitive loop） | 连续迭代尝试相似方向，收益递减 | 无法自行跳出局部最优 |
| **停顿**（Stalling） | 完成一块工作后输出摘要、等待用户反馈 | 会话看似存活，工作实际已停（比崩溃更常见） |
| **运行时脆弱性**（Runtime fragility） | 上下文压缩静默打断循环；关闭会话连带摧毁定时器 | 失败默认不被察觉 |

本框架中的每一个机制，都直接针对上述三种失败。

## 核心机制

```
    ┌── Orchestrator（当前会话 / 持久化 cron）──┐
    │ 监控状态文件 → 检测停顿 → 注入方向            │
    └────┬─────────────┬─────────────┬────────────┘
      [Task A]      [Task B]      [Task C]   ← 各自独立的全新会话
```

三条核心设计决策：

1. **执行与评估分离** —— 做工作的 Agent 不评判自身进展；停顿判定由编排层基于量化指标做出。
2. **全新会话优于 resume** —— 上下文累积是认知循环的首要成因。每次迭代都以全新上下文开始，状态通过文件注入。
3. **强制方向多样性** —— 每次迭代前读取已尝试的方向列表；新方向必须与全部历史记录不同。

## 核心特性

- 🔇 **零交互** —— 运行期间绝不向用户发问，自行解决歧义并记录决策日志。
- 💾 **状态文件持久化** —— 所有进展写入 `state/` 文件，而非对话记忆；每次迭代用全新会话。
- 📉 **停顿检测与强制转向** —— 量化指标判定停顿，停顿则改变结构性约束，而非战术参数。
- 💓 **三层心跳看门狗** —— L0 常驻 shell 守卫 / L1 持久化 cron / L2 业务循环，任一层死亡都能被另一层检测并恢复。
- 🛡️ **守护者/工作者分离** —— 心跳巡检对非自身任务只能做：存活检查、重启、推动；绝不越权读写数据或代为汇报。

## 快速开始

```bash
# 1. 初始化任务目录，写入 state/task_spec.md 和一个初始的 progress.json
mkdir -p {task}/state {task}/logs

# 2. 启动编排器循环（以 Claude Code 为例）
#    /loop 2h 检查所有任务：
#      (1) 读取 progress.json
#      (2) 若 stale_count>=3 则生成全新方向
#      (3) 通过 Agent 工具启动一个工作智能体（附带明确目标和完成标准）
#      (4) 将结果写回状态文件 —— 零交互

# 3. 注册持久化心跳看门狗（跨会话存活）
#    每小时巡检：写入时间戳 → 检查每个循环 last_seen → 超时重启 → 停顿则推动
```

> 完整协议细则（状态文件格式、子智能体调度模式、工程约束等）见 [skill.zh.md](./skill.zh.md)。

## 目录结构约定

```
{task}/
├── state/
│   ├── task_spec.md           # 目标 / 里程碑 / 成功标准
│   ├── progress.json          # {iteration, total_findings, status, stale_count}
│   ├── findings.jsonl         # 累积的发现（仅追加）
│   ├── directions_tried.json  # 已尝试的方向
│   └── iteration_log.jsonl    # 每次迭代的摘要
└── logs/
    ├── work.jsonl             # 由工作智能体写入；决策标记为 level=decision
    ├── orchestrator.jsonl     # 由编排器写入
    └── heartbeat.jsonl        # 由心跳看门狗写入
```

日志行统一格式：

```json
{"ts":"...", "source":"...", "level":"info|warn|error|decision", "event":"...", "detail":"..."}
```

## 关键概念速查

| 术语 | 含义 |
|------|------|
| **Orchestrator（编排器）** | 当前会话或持久化 cron，负责监控状态文件、检测停顿、向工作 Agent 注入方向 |
| **stale_count** | 连续无进展的迭代计数；达到阈值即触发强制转向 |
| **last_seen** | 每个循环最近一次「报活」时间戳；心跳看门狗据此判定是否超时 |
| **Nudge（推动）** | 心跳巡检对疑似停顿的任务注入的轻量提示；连续三次无进展则判定结构性卡死 |
| **Pivot（转向）** | 停顿后改变**结构性约束**（环境/框架），而非在既有框架内调参 |
| **Fresh session** | 每次迭代以全新会话开始，仅注入经筛选的状态，避免上下文累积导致的认知循环 |

## 行为约束

框架对 Agent 行为有五条硬性约束：

1. **零交互** —— 不使用 Plan Mode、不使用提问工具、不以问题收尾；自行解决歧义并写入日志（`level=decision`）。
2. **准备好即执行** —— 准备的目的就是执行；提交、重新提交、修复、启动监控都是常规操作，无需确认。
3. **回调即报活** —— 每次回调的第一个动作都是更新自身 `last_seen`，再检查存活，失败则立即重启。
4. **状态持久化到文件** —— 所有进展写入 `state/` 文件，绝不依赖对话记忆。
5. **守护者/工作者分离** —— 心跳巡检对非自身任务只能：存活检查、重启、推动。

## 停顿检测与转向规则

| 机制 | 规则 |
|------|------|
| 停顿检测 | 一次迭代产生 0 条新发现或某项指标下降 → `stale_count + 1` |
| 强制转向 | `stale_count >= 2` → 改变结构性约束；`>= 4` → 标记提请人工关注 |
| 方向多样性 | 新方向必须与每一条已尝试的不同；停顿后注入扰动策略 |
| 轮次上限 | 单个工作会话上限为 15 轮或 30 分钟 |

> 「转变结构，而非战术」——当任务在某个框架内反复停顿，决定性的增益通常来自纠正环境/结构性约束本身，而非更卖力地调参。

## 局限性

请理性看待本框架的能力边界：

1. **分数口径** —— 自评分数来自框架内的多人格模拟评审，仅在相同协议内纵向可比，**不是外部的质量声明**。
2. **不消除幻觉来源** —— 编造的引用和数据伪影源自 LLM 本身；框架将外部核查变成流程中的机械步骤，但**不消除错误来源**。
3. **依赖协议约束，而非模型自律** —— 职责分离靠协议约束保障；移除约束会让越权行为卷土重来。
4. **非即装即用的代码** —— 本仓库提供的是约定与协议，具体实现需采用者按自身环境落地。
