# Deli_AutoResearch

本 skill 是一个面向长时程自主任务（数天到数周）的协议框架。它不附带任何可执行代码；而是规定了一套经过实战检验的约定：如何持久化状态、如何检测停顿、如何分层设置守护机制，以及哪些约束规范着智能体的行为。实现细节留给采用者的运行环境自行决定。

## 1. 动机

长时间运行的代码智能体表现出三种反复出现的失败模式：

1. 认知循环——连续的迭代尝试相似的方向，收益递减，无法自行跳出局部最优。
2. 停顿——智能体完成一块工作后，输出一个摘要，然后等待用户反馈。从外部看会话似乎是存活的，轮询也在运行，但工作实际上已经停止。运行日志显示，这比崩溃更常见。
3. 运行时脆弱性——上下文压缩会静默地打断循环；关闭一个会话会连带摧毁寄生其上的定时器。失败默认情况下不会被察觉。

这三者的共同原因是缺失工程脚手架，而非模型能力不足。本框架中的每一个机制都针对上述失败模式。

## 2. 行为约束

1. 零交互——运行期间不得向用户发问：不使用 Plan Mode、不使用提问工具、不以问题收尾。持续工作直到用户让你停下。自行解决歧义，并将推理过程写入日志（level=decision）。
2. 准备好即执行——最常见的隐性违规：完成所有准备后却问「我应该提交吗？」。准备的目的就是执行；提交、重新提交、修复、启动监控都是常规操作，无需确认。
3. 回调即报活——上下文压缩后循环会静默死亡。每一次回调的第一个动作都是更新自身的 last_seen，然后检查存活状态；一旦检测到失败就立即重启并记录日志。
4. 状态持久化到文件——所有进展都写入 state/ 文件，而非对话记忆。每次迭代都以全新的会话开始，只注入经过筛选的状态；绝不使用 resume。
5. 守护者/工作者分离——心跳巡检对非自身所属的任务只能采取三种动作：存活检查、重启、推动（nudge）。它不读取这些任务的数据、不修改它们的状态文件、也不替它们向用户汇报。

## 3. 架构

    ┌── Orchestrator（当前会话 / 持久化 cron）──┐
    │ 监控状态文件 → 检测停顿 → 注入方向            │
    └────┬─────────────┬─────────────┬────────────┘
      [Task A]      [Task B]      [Task C]   ← 各自独立的全新会话

核心设计决策：
- 执行与评估分离——做工作的智能体不评判自身的进展；停顿判定由编排层基于量化指标做出。
- 全新会话优于 resume——上下文累积是认知循环的首要成因。每次迭代都以全新上下文开始；状态通过文件注入。
- 强制方向多样性——每次迭代前，读取已尝试的方向列表；新方向必须与全部历史记录不同。

## 4. 状态文件

    {task}/state/
    ├── task_spec.md           # 目标 / 里程碑 / 成功标准
    ├── progress.json          # {iteration, total_findings, status, stale_count}
    ├── findings.jsonl         # 累积的发现（仅追加）
    ├── directions_tried.json  # 已尝试的方向
    └── iteration_log.jsonl    # 每次迭代的摘要

    {task}/logs/
    ├── work.jsonl             # 由工作智能体写入；决策标记为 level=decision
    ├── orchestrator.jsonl     # 由编排器写入
    └── heartbeat.jsonl        # 由心跳看门狗写入

日志行格式：{"ts":"...", "source":"...", "level":"info|warn|error|decision", "event":"...", "detail":"..."}

## 5. 用法

    # 1. 初始化任务目录，写入 state/task_spec.md 和一个初始的 progress.json

    # 2. 启动编排器循环：
    /loop 2h 检查 : 下的所有任务：(1) 读取 progress.json；
    (2) 若 stale_count>=3 则生成一个全新方向；(3) 通过 Agent 工具
    启动一个工作智能体（附带明确的目标和完成标准）；
    (4) 将结果写回状态文件。零交互。

    # 3. 注册一个持久化心跳看门狗（跨会话存活）：
    每小时巡检：写入一个时间戳；将每个循环的 last_seen 与
    间隔×3 比对，超时则重启；检查每个任务 2 小时内的进展停顿，停顿则推动。
    零交互。

## 6. 停顿检测与转向

| 机制 | 规则 |
|-----------|------|
| 停顿检测 | 一次迭代产生 0 条新发现或某项指标下降 → stale_count + 1 |
| 强制转向 | stale_count >= 2 → 改变结构性约束，而非战术参数；>= 4 → 标记以提请人工关注 |
| 方向多样性 | 新方向必须与每一条已尝试的不同；停顿后注入一种扰动策略 |
| 轮次上限 | 单个工作会话上限为 15 轮或 30 分钟 |

「转变结构，而非战术」来自实践：当一个任务在某个框架内反复停顿时，决定性的增益通常来自纠正环境/结构性约束本身，而非在既有框架内更卖力地调参。

## 7. 心跳看门狗

业务循环本身是不可靠的，需要一个独立的守护层。三层相互校验（V3）：

| 层级 | 形式 | 依赖 | 职责 |
|-------|------|------------|------|
| L0 | 常驻 shell 守卫 | 无会话 | heartbeat 停顿 > 2h → 通过无头智能体拉起一次紧急巡检 |
| L1 | 持久化 cron，每小时 | 一个存活的交互式会话 | 检查每个循环的 last_seen，重启超时的循环，检测停顿并推动 |
| L2 | 业务循环 | 各自的会话 | 每次回调的第一行更新自身的 last_seen |

任一层死亡都能被另一层检测并恢复。

停顿检测：若进展超过 2 小时无更新，且最后输出是一个问题 → 判定为停顿，启动一个推动子智能体。连续三次推动仍无进展 → 判定为结构性卡死；停止推动并以一个新方向重新开始。2 小时的阈值被刻意设定得比 4 小时的卡死任务阈值更短。

## 8. 子智能体调度模式

| 模式 | 用途 | 核心思想 |
|---------|-----|----------|
| A 目标驱动 | 研究迭代 | 注入已尝试的方向，要求可验证的发现，写回 findings.jsonl |
| B 并行探索 | 复杂子问题 | 在一条消息中并发多个智能体：调研、反驳、跨领域类比 |
| C 实验运行 | 长时间计算任务 | 提交后立即启动分钟级轮询：自动诊断错误、修复、重新提交 |
| D 验证 | 迭代后 QA | 一个独立子智能体审计发现的证据链 |

一个子智能体的 prompt 应当包含：背景、一个可验证的交付物、工作目录、文件/行数上限，以及完成标准。

## 9. 工程约束

1. 每次迭代最多 5 个大文件；单个文件不超过 300 行。
2. 状态通过文件注入，而非对话历史。
3. 验证（test / compile / check）必须在迭代之间运行。
4. 引用类内容每 20 条核实一次，绝不积压批处理。
5. 面对多个候选方向时，优先增加多样性，而非对单一方向深挖。
6. 无法解决的外部依赖失败必须升级（完整报告 + 通知所有者 + 轮询等待回复）；绝不静默放弃。

## 10. 验证与局限

本框架已承载若干异构任务：学术论文写作、长时程研究等。论文赛道的产出：

| 论文 | 页数 | 引用数 | 自评 |
|-------|-------|-----------|------------|
| Autonomous Research Agents | 59 | 228 | 8.0/10 |
| Continual Learning | 65 | 326 | 8.0/10 |
| Long-Horizon Decision-Making | 55 | 384 | 8.0/10 |
| Self-Play（285B RL 实验 + 理论强化） | 75 | 217 | 8.6/10 |

局限：
1. 分数来自框架内的多人格模拟评审；仅在相同协议内纵向可比，并非外部的质量声明。
2. 有记录的最长连续运行为 72 小时，期间有 6 次方向性的人工输入——运维干预为零，方向性干预保留。
3. 编造的引用和数据伪影源自 LLM 本身；本框架将外部核查变成流程中的一个机械步骤，它并不消除错误来源。
4. 职责分离依赖协议约束，而非模型的自律；移除约束会让越权行为卷土重来。
